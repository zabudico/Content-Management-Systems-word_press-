## 5. Шаг 4. Базовое hardening

### Описание

Hardening — процесс уменьшения поверхности атаки путём отключения ненужных функций и ужесточения настроек. Выполнен ряд стандартных мер защиты.

### 4.1 Отключение редактора файлов из админки

В файл `wp-config.php` добавлена константа:

```php
define( 'DISALLOW_FILE_EDIT', true );
```

После добавления пункт меню **Внешний вид → Редактор** и **Плагины → Редактор** исчезли из административной панели. Это предотвращает возможность злоумышленника изменить PHP-файлы сайта через браузер даже при получении доступа к учётной записи администратора.

**Рисунок 5 — wp-config.php: добавлены константы DISALLOW_FILE_EDIT и WP_DEBUG**

![wp-config.php](img/wp_lab5_5.png)

### 4.2 Права на файлы и папки

Правильные права доступа предотвращают запись в файлы сервером в случае компрометации:

| Тип | Рекомендуемые права | Описание |
|-----|--------------------|----|
| Папки | `755` | Владелец: чтение/запись/выполнение; Остальные: чтение/выполнение |
| Файлы | `644` | Владелец: чтение/запись; Остальные: только чтение |
| wp-config.php | `640` | Только владелец и группа |

Проверка и установка прав выполнена через инструмент AIOS в разделе **Filesystem Security** (см. Шаг 5).

### 4.3 Защита wp-config.php через .htaccess

В файл `.htaccess` в корне WordPress добавлен блок:

```apache
# Защита wp-config.php
<files wp-config.php>
    order allow,deny
    deny from all
</files>

# Отключение просмотра директорий
Options -Indexes

# Защита .htaccess
<files .htaccess>
    order allow,deny
    deny from all
</files>
```

Это запрещает прямой HTTP-доступ к файлу конфигурации из браузера.

## 9. Ответы на контрольные вопросы

### Вопрос 1. Почему DISALLOW_FILE_EDIT и правильные права на wp-config.php существенно уменьшают риск пост-эксплойта?

**Сценарий без этих мер:**

Допустим, злоумышленник получил пароль администратора через фишинг или перебор. Имея доступ к админке, он может:
1. Открыть **Внешний вид → Редактор** и вставить PHP-шелл в файл темы (например, `functions.php`)
2. Получить полный контроль над сервером через веб-оболочку
3. Прочитать `wp-config.php` через браузер и получить пароль базы данных

**Как `DISALLOW_FILE_EDIT = true` снижает риск:**

Константа запрещает встроенный редактор файлов WordPress. Даже администратор не сможет редактировать PHP-файлы через браузер. Злоумышленник с украденным паролем теряет возможность быстро установить бэкдор через стандартный интерфейс.

```php
// После добавления этой строки:
define( 'DISALLOW_FILE_EDIT', true );

// Пункты меню «Внешний вид → Редактор» и «Плагины → Редактор» исчезают
```

**Как права на `wp-config.php` снижают риск:**

`wp-config.php` содержит критические данные: логин и пароль БД, секретные ключи, префикс таблиц. При правах `644` (по умолчанию) любой PHP-скрипт на сервере может прочитать этот файл. При правах `640` — только владелец и группа веб-сервера. Запрет через `.htaccess` (`deny from all`) предотвращает прямой HTTP-доступ к файлу.

**Итог:** оба механизма работают на этапе **пост-эксплойта** — когда первичный взлом уже произошёл, они ограничивают дальнейшее распространение атаки и повышение привилегий.

### Вопрос 2. Какие параметры Login Lockdown/Firewall выбраны и почему?

**Login Lockdown — выбранные параметры:**

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Max Login Attempts | **5** | Средний пользователь делает не более 2–3 попыток. 5 — достаточно для случайных ошибок (ошибка раскладки, Caps Lock), но критично мало для автоматизированного перебора (миллионы попыток/час) |
| Login Retry Period | **15 мин** | Окно в 15 минут: если 5 попыток за 15 минут — это атака, не случайность |
| Lockout Time | **30 мин** | Компромисс между безопасностью и UX: атакующий теряет полчаса, легитимный пользователь ждёт полчаса. Для production рекомендуется 60–120 мин |
| Generic Error Message | **Да** | Не раскрывает, что именно неверно. Сообщение «Неверный логин или пароль» лучше, чем «Пользователь не найден» — последнее подтверждает существование логина |

**Firewall — выбранные правила:**

Активирован **базовый уровень** (Basic Firewall), а не максимальный. Причина: агрессивные правила WAF могут ломать легитимный функционал плагинов. Стратегия «начать с базового, добавлять по необходимости» снижает риск ложных срабатываний.

**Баланс безопасность/UX:** выбранные параметры позволяют обычному пользователю комфортно работать (5 попыток — достаточно), при этом делают автоматизированный перебор практически невозможным (при скорости 1000 попыток/мин IP будет заблокирован через 5 мс после начала атаки).

### Вопрос 3. Чем отличаются меры защиты на уровне WordPress от мер на уровне веб-сервера и ОС?

**Три уровня защиты:**

```
┌─────────────────────────────────┐
│        Операционная система     │  ← Уровень 1
│   Права файлов, пользователи,   │
│   firewall (iptables/ufw)       │
├─────────────────────────────────┤
│         Веб-сервер (Apache)     │  ← Уровень 2
│   .htaccess, mod_security,      │
│   SSL/TLS, rate limiting        │
├─────────────────────────────────┤
│        WordPress + Плагины      │  ← Уровень 3
│   AIOS, роли, nonce, DISALLOW   │
└─────────────────────────────────┘
```

**Сравнительная таблица:**

| Аспект | WordPress/Плагин | Веб-сервер (Apache) | ОС |
|--------|-----------------|---------------------|----|
| Где работает | PHP/приложение | HTTP-слой | Ядро/система |
| Что защищает | Бизнес-логику, пользователей | HTTP-трафик, URL | Файлы, процессы, сеть |
| Пример | Login Lockdown блокирует брутфорс входа | `mod_evasive` ограничивает запросы на уровне Apache | `iptables` блокирует IP на уровне сетевого стека |
| Плюсы | Легко настраивается через UI, учитывает контекст WP | Работает до выполнения PHP, быстрее | Самый низкий уровень, не обходится через PHP |
| Минусы | PHP должен запуститься, можно обойти RCE | Не понимает WP-логику | Требует root-доступа, сложнее настроить |

**Вывод:** меры на уровне WordPress удобны и контекстуальны, но зависят от работы PHP. Если злоумышленник эксплуатирует уязвимость на уровне сервера или ОС, WordPress-плагины не помогут. Полноценная защита требует всех трёх уровней — принцип **«Defence in Depth»** (эшелонированная оборона).

### Вопрос 4. Что обязательно включать в «полный» бэкап WP и как проверить восстановление?

**Компоненты полного бэкапа WordPress:**

```
Полный бэкап = База данных + Файлы сайта
```

| Компонент | Что включает | Почему важно |
|-----------|-------------|-------------|
| **База данных (SQL)** | Все записи, страницы, пользователи, настройки, комментарии, метаданные | Всё содержимое сайта |
| **wp-content/themes/** | Активная и неактивные темы | Внешний вид сайта |
| **wp-content/plugins/** | Все установленные плагины | Функциональность |
| **wp-content/uploads/** | Изображения, документы, медиафайлы | Медиабиблиотека |
| **wp-config.php** | Настройки подключения к БД | Конфигурация |
| **.htaccess** | Правила сервера, постоянные ссылки | Маршрутизация |

**Что НЕ обязательно бэкапить:**
- `wp-includes/` и `wp-admin/` — они восстанавливаются переустановкой WordPress
- Кэш плагинов (`wp-content/cache/`) — пересоздаётся автоматически

**Как проверить, что восстановление работает:**

Правило: **бэкап, который не тестировался на восстановление, не является бэкапом**.

Процедура проверки (выполнена в данной работе):
1. Создать резервную копию БД через AIOS
2. Намеренно удалить данные (запись + медиафайл)
3. Восстановить БД через phpMyAdmin → импорт SQL
4. Проверить: удалённая запись присутствует 
5. Проверить: медиафайл в базе восстановлен 

**Периодичность проверки восстановления:** не реже 1 раза в квартал для production-сайтов. Рекомендуется автоматизировать тест-восстановление на staging-окружении.
